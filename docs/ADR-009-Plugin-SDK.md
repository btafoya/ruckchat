# ADR 009: Plugin SDK

## Status

Accepted

## Context

RuckChat needs a way for operators to extend the server without modifying core
source code. Examples include custom logging, moderation, slash commands, and
integration with external systems. The extension mechanism must be part of the
self-hosted server, not a separate SaaS marketplace, and should reuse the
existing domain model and services.

Key constraints:

- Plugins must be able to react to server events and inject custom behavior.
- The server and plugins must share the same `ruckchat-domain` types and event
  definitions to avoid drift.
- v1 does not require sandboxing or code signing; operators are responsible for
  only loading trusted plugins.
- The implementation must not add new infrastructure dependencies (no separate
  plugin registry, scheduler, or message queue).

## Decision

Introduce an in-process plugin SDK exposed by the `ruckchat-plugin-sdk` crate.
Plugins are native Rust `cdylib` crates that implement the `Plugin` trait and
export an entry point generated by the `declare_plugin!` macro. The server loads
them from a configured `plugin_dir` at startup, validates their API version,
multiplexes server events to them, and exposes an HTTP endpoint for slash-style
commands.

Specific choices:

1. **SDK crate**: `crates/ruckchat-plugin-sdk` defines `Plugin`, `HostApi`,
   `PluginEvent`, `PluginCommand`, `CommandResponse`, and the `declare_plugin!`
   macro. Plugin authors depend only on this crate and the shared domain crates.

2. **Cross-DLL interface**: The macro generates a single `ruckchat_plugin_entry`
   symbol that returns a `PluginExports` descriptor containing `create` and
   `destroy` function pointers. `create` returns a `*mut PluginInstance` wrapping
   a `Box<dyn Plugin>`. This relies on the server and plugin using the same Rust
   compiler and SDK version; the API-version check catches incompatible plugins.

3. **Host API**: The server implements `HostApi` with synchronous methods
   (`log`, `get_config`, `get_user`, `get_channel`, `send_message`, `emit_event`).
   The implementation uses `tokio::task::block_in_place` +
   `Handle::current().block_on` to call the existing async services and
   repositories, keeping the author-facing API simple.

4. **Server loader and manager**: `server/src/plugins/loader.rs` uses
   `libloading` to open trusted dynamic libraries. `manager.rs` owns loaded
   plugins behind a mutex, initializes them with per-plugin TOML configuration,
   dispatches events, and routes commands.

5. **Event multiplexing**: `server/src/plugins/bus.rs` provides
   `CompositeEventBus`, which implements the existing `EventBus` trait. It first
   publishes to the WebSocket event bus and then delivers `MessageReceived`,
   `MessageUpdated`, and `MessageDeleted` plugin events. Services and tests do
   not change their event-bus interface.

6. **Command endpoint**: `POST /plugins/{plugin}/commands/{command}` accepts the
   target conversation and arguments, dispatches to the named plugin, and returns
   a `CommandResponse` (message, ephemeral, or error). Full slash-command parsing
   in the desktop composer is a later client-side enhancement.

7. **Configuration**: Each plugin reads its configuration from
   `PLUGIN_DIR/<plugin-name>.toml` during initialization. The configuration is
   passed as a `serde_json::Value`.

## Consequences

- Plugin authors write normal Rust code against a small SDK and can reuse
  `ruckchat-domain` entities.
- The server remains a single process; no additional runtime infrastructure is
  needed.
- Because plugins share the server address space, a buggy plugin can crash the
  server. v1 accepts this trade-off and documents the trust boundary.
- Plugins must be compiled for the same platform and with a compatible Rust/SDK
  version as the server.
- Future versions can add sandboxing, code signing, hot reload, or an
  out-of-process plugin runtime without changing the high-level SDK design.
