# 011 - MCP

## Purpose

RuckChat exposes a Model Context Protocol (MCP) server so that AI assistants can read context from and perform actions on behalf of authorized users. The MCP server reuses the same Rust server process and database; it is not a separate service.

## Transport

- v1 uses the Streamable HTTP transport from the `rmcp` Rust SDK.
- Endpoint: `/mcp/v1/sse` on the same host as the REST API.
  - `POST` requests carry JSON-RPC client messages and return responses as SSE events.
  - `GET` requests open a long-lived SSE stream for server-initiated messages when the client sends a session ID in the `Mcp-Session-Id` header.
- Authentication uses the same session cookie or `Authorization: Bearer` header as the REST API and WebSocket endpoints.
- After a successful `initialize` call the server returns a `Mcp-Session-Id` header that the client must include on subsequent requests.

## Scope

The MCP server exposes only what the authenticated user is allowed to see and do:

- Read channels and direct messages the user can access.
- Post messages on behalf of the user, with explicit confirmation by default.
- Search message history within allowed scopes.
- List organization members and channels.

## Tools

| Tool | Input | Behavior |
|------|-------|----------|
| `list_channels` | `{ organization_id: uuid }` | List channels visible to the user. |
| `list_direct_messages` | `{ organization_id: uuid }` | List DM conversations for the user. |
| `get_messages` | `{ conversation_id: uuid, conversation_type: "channel" \| "direct_message", limit?: number, offset?: number }` | Fetch recent messages. |
| `search_messages` | `{ organization_id: uuid, query: string, limit?: number, offset?: number }` | Search messages with PostgreSQL full-text search. |
| `post_message` | `{ conversation_id: uuid, conversation_type: "channel" \| "direct_message", content: string, parent_id?: uuid, confirmed?: boolean }` | Post a message; requires `confirmed: true` when confirmation is enabled. |
| `get_user_profile` | `{ user_id: uuid }` | Read a user profile if visible. |

### Confirmation flow

When `MCP_REQUIRE_CONFIRMATION` is `true` (the default), `post_message` returns
a confirmation-pending response unless the request includes `confirmed: true`.
The response contains the planned `conversation_id`, `conversation_type`, and
`content` so the client can prompt the user and retry. This is a stateless v1
guard; future versions may use MCP sampling for a richer consent flow.

## Resources

- `ruckchat://organization/{id}` — organization metadata.
- `ruckchat://channel/{id}` — channel metadata.
- `ruckchat://conversation/{id}` — channel or DM metadata.
- `ruckchat://message/{id}` — single message.

## Permissions

- The MCP server calls the same service layer as the REST API.
- It does not bypass authorization or expose admin-only data.
- AI clients cannot delete messages or access files through MCP tools.

## Rate Limits and Safety

- MCP endpoints share the same rate limits as REST.
- `post_message` is logged and attributed to the user, not to the AI model.
- Sensitive operations (e.g., deleting content) are not exposed through MCP.

## Configuration

| Variable | Default | Description |
|----------|---------|-------------|
| `MCP_ENABLED` | `true` | Enable the MCP endpoint. |
| `MCP_REQUIRE_CONFIRMATION` | `true` | Require user confirmation before posting messages. |

## Testing

- Unit tests in `server/src/services/mcp.rs` exercise `McpService` against
  in-memory mock repositories, including authorization, confirmation flow, and
  error handling.
- Integration tests in `server/tests/mcp.rs` open a Streamable HTTP session
  through the Axum router, call `initialize`, and verify `tools/list`,
  `tools/call`, and `resources/read` end-to-end against PostgreSQL.

## Future Directions

- Add stdio transport for local MCP clients.
- Expose more granular tools for administration and analytics.
- Support tool result streaming for long-running searches.
- Use MCP sampling for message confirmation instead of a boolean flag.
