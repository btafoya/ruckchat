# 012 - Plugin SDK

## Purpose

The plugin SDK allows third-party developers to extend RuckChat with native Rust plugins that run in-process. Plugins can react to events, add commands, and integrate with external systems without modifying the server.

## Plugin Model

- Plugins are compiled as native dynamic libraries (`cdylib`).
- The server loads plugins from `plugin_dir` at startup. The default directory is `./plugins`, overridable via the `RUCKCHAT_PLUGIN_DIR` environment variable or `ruckchat.toml`.
- Each plugin exports a single `ruckchat_plugin_entry` symbol generated by the `declare_plugin!` macro. The symbol returns a `PluginExports` descriptor containing `create` and `destroy` function pointers.
- Plugins run in the same process as the server; a misbehaving plugin can crash the server, so plugins are loaded only from trusted sources.

## SDK Crate

The `ruckchat-plugin-sdk` crate provides:

- `Plugin` trait with `info`, `initialize`, `on_event`, `on_command`, and `shutdown` hooks.
- `PluginEvent` (message created/updated/deleted) and `PluginCommand` types.
- `HostApi` for logging, configuration access, database reads, sending messages, and emitting events.
- `CommandResponse` for replying to slash commands.
- The `declare_plugin!` macro for exporting the plugin entry point.

## Lifecycle

1. **Load** ‚Äî the server opens the dynamic library and calls the entry point.
2. **Initialize** ‚Äî the plugin receives a `serde_json::Value` configuration parsed from `plugin_dir/<plugin-name>.toml`.
3. **Run** ‚Äî the plugin responds to events and commands.
4. **Shutdown** ‚Äî the plugin is notified before the server exits via `shutdown`.

## Hooks

| Hook | Trigger | Use Case |
|------|---------|----------|
| `on_event(PluginEvent::MessageReceived)` | A message is created. | Filtering, moderation, external logging. |
| `on_event(PluginEvent::MessageUpdated)` | A message is updated. | Update propagation, indexing. |
| `on_event(PluginEvent::MessageDeleted)` | A message is deleted. | Cleanup, audit logging. |
| `on_command(PluginCommand)` | A user invokes a plugin command. | Custom slash commands. |

## Commands

- Commands are invoked over HTTP via `POST /plugins/{plugin}/commands/{command}`.
- The request body is `InvokeCommandRequest`, containing the target `conversation_id` and a `serde_json::Value` `arguments` object.
- The plugin receives a `PluginCommand` and returns a `CommandResponse`:
  - `Message { content }` ‚Äî a normal message reply.
  - `Ephemeral { content }` ‚Äî a private reply to the caller.
  - `Error { message }` ‚Äî a client-visible error.

## Configuration

- Each plugin reads configuration from `plugin_dir/<plugin-name>.toml` during initialization.
- Configuration is passed as a `serde_json::Value`; the plugin author decides its schema.
- Runtime configuration reloading is a post-MVP enhancement.

## Host API

Plugins interact with the server through a controlled `HostApi`:

- `log(level, message)`
- `get_config() -> serde_json::Value`
- `get_user(user_id) -> Option<User>`
- `get_channel(channel_id) -> Option<Channel>`
- `send_message(request) -> Result<Message, String>`
- `emit_event(event)`

Plugins do not execute raw SQL or access the database connection pool directly. The synchronous `HostApi` implementation uses `tokio::task::block_in_place` to call the server's existing async services and repositories.

## Sandboxing and Security

- v1 does not sandbox plugins. Operators must trust the plugins they install.
- Plugin binaries should be signed in post-MVP releases.
- The server validates the plugin's `api_version` against the SDK's `API_VERSION` and rejects incompatible plugins.

## Distribution

- Plugins are distributed as precompiled binaries for supported platforms.
- A plugin manifest (`plugin.toml`) may declare name, version, API version, and supported platforms in post-MVP releases.
- Community plugins are documented in the project wiki; official plugins live in the `ruckchat-plugins` repository.
